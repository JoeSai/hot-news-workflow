## 一、用户背景

### 核心画像

一位同时经营**珠宝店铺**和 **AI 科技自媒体号**的运营者。珠宝是主业，AI 自媒体是试验田——希望先用 AI 赛道账号验证这款工具的运营效果（能否真实涨粉、提升内容效率），跑出令人满意的成绩后再考虑将工具引入珠宝等其他赛道。

### 关键约束

| 约束 | 说明 |
|------|------|
| **不做自动发布** | 小红书等平台对 AI 托管发布敏感，账号有封禁风险。用户明确要求：**工具只生成草稿/参考内容，由用户手动发布** |
| **先验证再扩展** | 用户不会一上来就在珠宝号使用，需要在 AI 号上看到实际增长数据后才会信任工具 |
| **效果可衡量** | 用户需要清晰看到「用工具 vs 不用工具」的效率差异和涨粉差异 |

### 用户的实际工作流（当前手动）

```
刷各平台找热点 → 筛选 AI 相关选题 → 查资料写文案 → 做封面图 → 手动发布小红书 → 看数据
      ↑                                                                    ↓
      └──────────────── 根据数据反馈调整选题方向 ←──────────────────────────┘
```

**痛点集中在前三步**：找热点耗时、选题靠直觉、写文案效率低。

## 二、产品现状评估

### 当前能力

```
热点抓取（7+ 新闻源） → 关键词提取（YAKE/TF-IDF/TextRank） → 词云 / 热词列表 / 新闻详情
```

### 核心问题

**产品在「关键词提取」之后断裂了。** 用户拿到一堆关键词后，不知道下一步该怎么做——选哪个写？怎么写？写出来什么样？这导致工具只完成了工作流的 30%，用户在最需要帮助的环节（选题决策、内容生成）反而要回到纯手工模式。

### 信息源错位

当前新闻源（网易、澎湃、搜狐、腾讯等）偏**综合时政**，AI 赛道博主需要的是 36kr、量子位、机器之心、Hacker News 等**科技垂直源**。用户每次抓到的热点大量与 AI 无关，需要人工过滤，反而增加了工作量。

## 三、多用户视角分析

### 👤 视角 A：起号新手（粉丝 < 500）

> "我不知道今天该写什么，能不能直接告诉我？"

- 词云和关键词对我太抽象，我需要的是**选题建议**——「今天推荐你写这 3 个方向」
- 帮我出一份**草稿**就行，我自己改改润色后发布
- 最好有**小红书爆款标题参考**，我不会起吸引人的标题

### 👤 视角 B：成长期博主（粉丝 2k-1w，日更 3 个月）

> "效率是我的核心需求，一篇笔记从选题到成稿要压缩到 30 分钟内。"

- 需要**工作流模板**，一键跑完「抓取→提取→生成草稿」
- 需要**批量操作**——一次生成 3-5 篇不同选题的草稿，我挑最好的发
- 草稿要有**小红书风格**（emoji、分段、口语化），不要生硬的新闻稿

### 👤 视角 C：有变现需求的运营者（粉丝 5w+）

> "我要看到数据证明这个工具有用，不然我不会投入时间。"

- 需要**效果追踪**——用工具辅助的笔记 vs 纯手写笔记，互动数据对比
- 需要**热度趋势图**——帮我判断一个话题是上升期还是已经过气
- 如果 AI 号验证成功，我会考虑在**珠宝号**也使用，所以工具需要支持**多赛道/多账号场景**

### 👤 视角 D：技术型创作者（程序员转自媒体）

> "架构不错，但节点太少，我想自己加数据源。"

- 给我**API 文档**，让我能用脚本批量调用
- 支持**定时任务**，每天早 8 点自动抓取，我上班前就能看到今日选题
- 让我能**添加自定义爬虫**，比如抓 GitHub Trending 或 X 上的 AI 讨论

## 四、开发建议（按优先级排序）

> 核心原则：**工具只生成草稿和参考，不做自动发布。** 一切发布动作由用户手动完成。

### 🔴 P0 — 打通核心闭环（当前最紧迫）

#### P0.1 AI 内容生成节点（草稿模式）

**这是整个产品从「玩具」变成「工具」的关键。**

- 输入：热点关键词 + 相关新闻摘要
- 输出：小红书风格的**内容草稿**（而非成品），包含：
  - 标题（2-3 个备选）
  - 正文（口语化、带 emoji、带分段）
  - 推荐标签（5-8 个）
  - 封面文案建议
- 接入方式：支持 OpenAI / Claude / 国产大模型（通义千问、DeepSeek 等）API
- **明确标注「草稿仅供参考，请修改后手动发布」**
- 允许用户在节点内**直接编辑草稿**，调整后一键复制

```
设计要点：
- 草稿面板支持实时编辑（contentEditable 或 textarea）
- 提供「重新生成」按钮，可微调 prompt
- 支持选择文案风格：科普向 / 观点向 / 教程向 / 测评向
- 复制按钮分别复制：标题、正文、标签（方便粘贴到小红书 APP）
```

#### P0.2 AI 垂直信息源

当前新闻源对 AI 赛道博主价值有限，需要新增：

| 优先级 | 数据源 | 说明 |
|--------|--------|------|
| 必须 | **36kr AI 频道** | 国内 AI 产业最主流信息源 |
| 必须 | **量子位** | AI 技术 + 产品动态 |
| 高 | **机器之心** | 偏学术和前沿技术 |
| 高 | **Hacker News** | 全球科技社区热议话题 |
| 中 | **Product Hunt** | AI 新产品发现 |
| 中 | **GitHub Trending** | 开源项目趋势 |

#### P0.3 选题评分/推荐节点

在关键词提取之后、内容生成之前，增加「选题推荐」节点：

- 对提取的关键词做**赛道相关性评分**（AI 相关度 0-100）
- 结合**热度权重 + 时效性 + 赛道匹配度**排序
- 输出 TOP 3-5 推荐选题，附简要推荐理由
- 让用户点击选题后直接流入内容生成节点

### 🟠 P1 — 效率飞轮（让工具真正省时间）

#### P1.1 工作流模板

预设 3-5 个适合 AI 赛道的模板，新用户一键加载即可运行：

| 模板名称 | 预设流程 |
|----------|----------|
| AI 热点日报 | 科技源抓取 → 关键词提取 → 选题推荐 → 生成 3 篇短草稿 |
| AI 产品测评 | Product Hunt 抓取 → 筛选 AI 产品 → 生成测评草稿 |
| 技术科普 | Hacker News + 机器之心 → 提取技术关键词 → 生成科普草稿 |
| 行业周报 | 全源抓取 → 30 天趋势对比 → 生成周报框架 |

#### P1.2 一键执行工作流（Run All）

当前每个节点需要手动点击执行，中断了流程的连贯性。增加：

- 画布顶部「▶ 一键执行」按钮，从源头到末尾自动依次执行
- 执行过程中每个节点高亮显示当前状态（等待/运行中/完成/错误）
- 某节点失败时，标记错误并允许从该节点重新执行

#### P1.3 草稿管理与导出

- **草稿历史**：每次生成的草稿保存到后端，可回溯查看
- **导出格式**：Markdown / 纯文本 / JSON
- **批量生成**：对 TOP 5 选题一次性生成 5 篇草稿，用户挑选最佳

#### P1.4 后端持久化

用 SQLite 替代 localStorage，支持：
- 工作流配置持久化
- 抓取历史记录存储（为 P2 的趋势分析打基础）
- 草稿存储与管理

### 🟡 P2 — 数据驱动运营（帮用户做决策）

#### P2.1 热度趋势图

- 存储每日抓取的关键词数据

- 在草稿节点内增加「小红书样式预览」

- 让用户在工具内就能直观感受发出去的效果
- 预览页提供「复制标题」「复制正文」「复制标签」快捷按钮

#### P2.4 封面图辅助

- 提供小红书常用封面模板（文字卡片、对比图、清单图）
- 用户输入标题文案，自动渲染为封面图预览
- 支持导出为图片，用户下载后手动上传到小红书

## 五、设计红线

以下是产品设计中**必须遵守的原则**：

1. **绝不自动发布** — 所有内容必须经过用户手动审核和修改后，由用户自行发布。工具的定位是「智能助手」而非「自动驾驶」
2. **草稿即参考** — UI 上明确标注生成内容为「草稿 / 参考」，引导用户修改和个性化，避免千篇一律的 AI 味
3. **数据最小化** — 不存储用户的小红书账号密码、不调用小红书私有 API、不模拟登录行为
4. **效果可验证** — 提供数据回填和对比机制，让用户能客观评估工具价值
5. **赛道可迁移** — 架构设计考虑未来扩展到其他赛道（珠宝、美妆、教育等），信息源和文案风格模块化

#### 完成情况

| P0.2 AI 垂直信息源 | ✅ 完成 | 36kr、量子位、机器之心、Hacker News + 微博/知乎/今日头条，**超预期** |
| P0.3 选题评分/推荐 | ✅ 完成 | AI 关键词库评分 + 类型标签（科普/观点/教程/测评），推荐逻辑合理 |

**用户体验反馈：** 现在可以从头到尾跑通「抓 AI 热点 → 提取关键词 → 推荐选题 → 生成草稿 → 复制发布」这条链路了。作为一个日更博主，这个闭环对我来说是 **从 0 到 1 的变化**——之前这个工具只能帮我看热点，现在可以帮我出草稿了。

#### 仍需改进的使用体验问题

1. **每个节点要手动点一次**——抓完热点要点提取、提取完要点推荐、推荐完要点生成，连续 4 次点击太碎了
2. **API Key 每次重启都要重新输入**——存在 localStorage 里的工作流不保存 API Key（出于安全考虑可以理解，但至少给个环境变量方案）
3. **草稿无法保存历史**——我昨天生成的草稿，今天就找不到了
4. **选题推荐是前端纯算法**——基于关键词列表匹配，对于新出的 AI 产品名（如新发布的模型）无法识别

#### 技术架构评审

作为技术架构师，我对当前代码做如下评审：

##### 做得好的地方

- **数据通过 React Flow 边传递**，节点间松耦合，新增节点不影响已有节点
- **爬虫基类 BaseSpider** 抽象合理，UA 轮换 + 退避重试策略完善
- **YAKE 短语提取** 比纯分词能生成多词短语，但对中文分词质量差（详见 v0.3 评审），需要用 jieba 预分词修复
- **前后端职责分离**清晰：前端负责工作流编排，后端负责数据采集和 AI 调用

##### 需要注意的架构问题

**🔴 安全问题：API Key 明文传输**

```
当前：前端输入 API Key → HTTP POST body → 后端转发到 LLM API
问题：API Key 在网络传输中明文暴露，localStorage 里也可能留痕
```

建议方案：
- 后端通过环境变量 `DEEPSEEK_API_KEY` / `OPENAI_API_KEY` 读取，前端不传 Key
- 后端已经有 `default_key_env` 的设计（在 `API_CONFIGS` 里），但前端没有利用这个降级路径
- 短期：前端增加「使用服务端 Key」的选项；长期：API Key 只存后端

**🟠 架构瓶颈：爬虫全在 spider.py 一个文件**

7 个 AI 垂直源（36kr/量子位/机器之心/HN/微博/知乎/今日头条）都写在 `spider.py` 里（几百行），而原有的通用新闻源在 `server/crawlers/` 下是模块化的。这导致：
- 新增爬虫只能改 spider.py，冲突风险高
- 无法单独测试某个爬虫
- 代码导航困难

建议：将每个爬虫平台拆到 `server/crawlers/` 下独立文件，统一继承 `BaseSpider`。

**🟠 前端状态膨胀风险**

```typescript
// useWorkflowStore.ts - 每个节点的 data 包含完整的 news[] 和 keywords[]
// 50 条新闻 × 7 个平台 = 350 条 → 序列化到 localStorage 每次操作都 JSON.stringify 全量
```

当数据量增大时（多平台大批量抓取），localStorage 的 5MB 限制会被触及，且每次节点交互都触发全量序列化。

建议：
- 大数据（news[]、keywords[]）存后端或 IndexedDB，节点 data 只存引用 ID
- localStorage 只存工作流结构（nodes/edges 位置和配置），不存数据负载

**🟡 爬虫同步阻塞**

`spider.py` 中的 `crawl_all` 虽然被 FastAPI 标记为 `async`，但内部实际使用 `requests`（同步库）+ `time.sleep`，会阻塞事件循环。在并发请求场景下会卡住整个服务。

建议：
- 用 `asyncio.to_thread()` 包装同步爬虫调用
- 或迁移到 `httpx.AsyncClient`

**🟡 草稿解析用正则太脆弱**

`ContentGenerateNode.tsx` 的 `parseDraft()` 用正则解析 LLM 输出的 Markdown，但 LLM 输出格式不稳定（有时用 `###` 有时用 `##`，有时标题带序号有时不带）。

建议：
- 让后端的 prompt 加强格式约束（用 JSON 输出模式或 XML 标签包裹）
- 或者后端解析好再返回结构化数据 `{ titles: [], body: "", tags: [] }`，前端不再负责解析

#### 更新后的优先级

#### 新发现的严重问题：关键词提取质量

在实际使用中发现 **YAKE 对中文标题的提取效果很差**，这直接影响选题推荐和内容生成的输入质量。

##### 问题复现

标题：`如何评价Anthropic封杀openclaw，将其提出claude订阅服务的白名单？`

YAKE 提取的 TOP 1 结果：`，将其提出claude订阅服务的白名单`

**问题根因：YAKE 是为欧洲语言设计的关键词提取器，它的分词器不认识中文标点。** 逗号 `，`、问号 `？` 等会被当作普通字符粘连到词语上，导致提取出包含标点的垃圾短语。用 `lan="zh"` 参数只是告诉 YAKE 用中文停用词，并不会改善分词质量。

##### 验证实验

我对当前代码做了 5 种方案的对比测试：

| 方案 | 方法 | 效果 |
|------|------|------|
| 当前代码 | 原始 YAKE | ❌ 提取出 `，将其提出claude订阅服务的白名单` |
| 方案 1 | 清理标点后 YAKE | ❌ 仍出现 `将其提出 claude 订阅服务的白名单`（语义断裂） |
| 方案 2 | jieba 预分词 → YAKE | ✅ `Anthropic 封杀 openclaw`、`白名单`、`claude` |
| 方案 3 | jieba TF-IDF（单词） | ✅ `Anthropic`、`openclaw`、`claude`、`白名单`、`封杀` |
| 方案 4 | jieba 分词 + 滑动窗口短语 | ✅✅ `Anthropic封杀`、`封杀openclaw`、`claude订阅`、`订阅服务`、`白名单` |

**结论：方案 2（jieba 预分词 → YAKE）或方案 4（jieba + 滑动窗口）最优。**

##### 建议的修复方案

**推荐：jieba 预分词 + 标点作为短语边界 + YAKE**

核心思路：用 jieba 做中文分词（它擅长这个），然后用空格拼接分词结果喂给 YAKE（它擅长多词短语排序和去重）。中文标点替换为句号，作为 YAKE 的句子边界，防止跨标点组合短语。

```python
def preprocess_for_yake(text):
    """jieba 分词预处理，解决 YAKE 中文分词差的问题"""
    import jieba
    import re
    words = jieba.lcut(text)
    segments = []
    for w in words:
        w = w.strip()
        if not w:
            continue
        # 中英文标点 → 句子边界（YAKE 用句号分割短语范围）
        if re.match(r'^[，。？！、；：""''【】《》（）\[\]{}…—\-·,.?!;:%]+$', w):
            segments.append('.')
        else:
            segments.append(w)
    text = ' '.join(segments)
    text = re.sub(r'\s+', ' ', text).strip()
    return text
```

改动点：在 `extract_phrases_yake()` 函数中，将 `keywords = kw_extractor.extract_keywords(title)` 改为 `keywords = kw_extractor.extract_keywords(preprocess_for_yake(title))`。

**额外建议：增加后过滤规则**

即使预处理后，YAKE 仍可能产出低质量短语。建议增加：

```python
# 过滤规则
def is_valid_keyword(word):
    # 1. 过滤纯数字/版本号
    if re.match(r'^[\d.%]+$', word):
        return False
    # 2. 过滤过长的短语（>15字大概率是垃圾）
    if len(word) > 15:
        return False
    # 3. 过滤以停用词开头/结尾的短语
    for sw in ['的', '了', '是', '在', '将', '其', '把', '被']:
        if word.startswith(sw) or word.endswith(sw):
            return False
    return True
```

### v0.4+ 评审：关键词与热点脱节问题（架构级）

#### 问题描述

用户反馈：提取到的热词如「乒乓球」「世界杯」「市场化」，直接用来生成内容，产出的文案**只和关键词字面含义有关，和背后的热点事件毫无关系**。

这不是一个 bug，而是**当前数据流架构的根本性缺陷**。

#### 问题根因分析

当前数据流在关键词提取环节发生了**语境断裂**：

```
┌─ 新闻标题 ─────────────────────────────────────────┐
│  "中国乒乓球队包揽世锦赛五金，创历史最佳战绩"        │
│  "世界杯预选赛：中国队客场2-1逆转泰国"               │
│  "央行推进利率市场化改革，LPR机制迎重大调整"           │
└────────────────────┬───────────────────────────────┘
                     ↓ 关键词提取（只保留词，丢失语境）
┌─ 提取的关键词 ─────┐
│  乒乓球  weight=0.8 │ ← 丢失了"包揽世锦赛五金"
│  世界杯  weight=0.7 │ ← 丢失了"中国队逆转泰国"
│  市场化  weight=0.6 │ ← 丢失了"LPR机制改革"
└────────┬───────────┘
         ↓ 用户选择"乒乓球" → 内容生成
┌─ 生成的 prompt ──────────────────────────────────────┐
│  热词：乒乓球                                         │
│  参考新闻：(前5条新闻，大概率和乒乓球无关)              │
│  → AI 写了一篇泛泛而谈乒乓球运动的科普文 ❌            │
│  → 用户想要的是：关于"包揽世锦赛五金"这个热点的文案 ✅   │
└──────────────────────────────────────────────────────┘
```

具体来说有**两层脱节**：

1. **关键词丢失事件语境** — `Keyword` 接口只有 `{ word, weight }`，没有回溯到来源新闻的能力。「乒乓球」这个词被从「中国乒乓球队包揽世锦赛五金」中剥离出来后，就失去了事件信息
2. **新闻上下文匹配错误** — 内容生成节点的 `inputNews.slice(0, 5)` 取的是前 5 条新闻标题，不是与选中关键词相关的新闻。如果用户选了「乒乓球」，但前 5 条新闻是关于 AI 的，prompt 中的新闻参考和关键词对不上号

#### 这个问题有多严重？

**非常严重。这是当前产品最大的使用价值瓶颈。**

对比两种生成效果：

| | 脱节的（当前） | 关联的（期望） |
|---|---|---|
| Prompt | `热词：乒乓球` | `热点事件：中国乒乓球队包揽世锦赛五金，创历史最佳战绩` |
| 生成结果 | 乒乓球运动入门指南 | 五金包揽！中国乒乓天团为什么这么强？| 
| 蹭到热点？ | ❌ 没蹭到 | ✅ 精准蹭到 |
| 用户还需要改？ | 几乎要重写 | 润色即可 |

**如果生成的内容蹭不到热点，那整条工作流（热点抓取 → 关键词 → 生成）就失去了意义**——用户不如直接让 AI 随便写一篇。

#### 推荐的解决方案

##### 方案 A：关键词携带来源新闻（推荐，最小改动）

**核心思路：让每个关键词知道自己从哪些新闻标题中提取出来的。**

1. **数据模型扩展** — `Keyword` 增加 `sourceNews` 字段：

```typescript
// types/workflow.ts
interface Keyword {
  word: string;
  weight: number;
  type?: 'word' | 'phrase';
  sourceNews?: string[];  // 🆕 包含该关键词的原始新闻标题
}
```

2. **后端提取时记录来源** — `extract_phrases_yake()` 在提取后做反向匹配：

```python
# 对每个提取出的关键词，找到包含它的原始标题
for kw_item in result:
    kw_item["source_news"] = [
        title for title in titles 
        if kw_item["word"].lower() in title.lower()
    ][:5]  # 最多保留5条来源
```

3. **内容生成节点改进** — 生成时用选中关键词的 `sourceNews` 作为上下文，而非前 5 条无关新闻：

```python
# 后端 prompt 改进
prompt = f"""
热点事件：
{chr(10).join(relevant_news_titles)}

请围绕以上热点事件，结合关键词「{keywords_str}」，生成一篇小红书风格的内容草稿...
"""
```

**改动影响范围：**
- 后端 `extract_keywords` → 增加 source_news 字段
- 后端 `generate_content` → prompt 改用 relevant news
- 前端 `Keyword` 类型 → 增加 sourceNews
- 前端 `ContentGenerateNode` → 传递 selectedKeywords 的 sourceNews
- **不需要改数据流架构，不需要加新节点**

##### 方案 B：把选题推荐从「关键词评分」改为「新闻事件评分」（更彻底）

将选题推荐的粒度从「关键词」提升为「新闻事件」：

- 选题推荐节点输入的不是 `Keyword[]`，而是 `NewsItem[]`
- 对每条新闻做 AI 赛道相关性评分
- 用户选择的是「一条具体的热点新闻」，而非「一个脱离上下文的关键词」
- 内容生成节点直接围绕这条新闻的完整标题 + URL 来生成

这个方案更彻底，但改动大，建议作为 P2 考虑。

##### 方案对比

| | 方案 A：关键词携带来源 | 方案 B：新闻事件为中心 |
|---|---|---|
| 改动大小 | 小，只加字段 + 改 prompt | 大，重构选题推荐逻辑 |
| 效果 | 好，prompt 有正确上下文 | 最好，从根源解决 |
| 建议 | **P1 立即做** | P2 再考虑 |
